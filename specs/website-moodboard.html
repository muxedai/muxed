The problem with MCP at scale

Every MCP server you connect dumps its full tool schemas into the model's context window. More tools means more tokens burned before work even starts — and worse accuracy as the model struggles to pick the right one.

token overhead

Anthropic found intermediate results flow through the model unnecessarily, wasting almost all tokens.

context consumed

A standard 3-4 server setup fills a quarter of the context before the agent starts working.

3-4
server limit

Tool selection accuracy collapses after just a handful of connected MCP servers.

More tools = worse results. Every token spent on tool schemas is a token not spent on reasoning, skills, or your actual task.

Code blocks: Two side-by-side examples now. First shows on-demand discovery (grep + info + call). Second shows chaining — piping mcpd calls through scripts without the LLM seeing intermediates.

Discover, don't preload

Instead of dumping 30,000 tokens of tool schemas into every prompt, agents search for what they need on demand. Find a tool, inspect its schema, call it — only the result enters the model's context.

terminal 
 1 # discover — not preload
 2 $ mcpd grep "read"
 3   filesystem/read_file      Read file contents
 4   postgres/query            Read data via SQL
 5
 6 # inspect only the tool you need
 7 $ mcpd info filesystem/read_file
 8   path (string, required)   File path to read
 9
10 # call it — only the result enters the LLM
11 $ mcpd call filesystem/read_file \
12     '{"path": "/tmp/config.json"}'

Chain outside the model

Pipe tool outputs through scripts, jq, or other mcpd calls. Intermediate data stays in the shell — the LLM only sees the final result. Same insight as Anthropic's code execution approach and Cloudflare's Code Mode, but as a simple CLI.

terminal — chaining 
1 # query DB, filter with jq, write to file
2 # model never sees the raw 10k-row result
3 $ mcpd call postgres/query \
4     '{"sql":"SELECT * FROM users"}' --json \
5   | jq '[.[] | select(.active)]' \
6   | mcpd call filesystem/write_file \
7     '{"path":"/tmp/active.json"}' -
8
9   Wrote 847 active users to /tmp/active.json

Real-world example: churn analysis

Your agent writes a script that chains MCP calls across PostHog and Intercom. It finds churned customers, pulls their support history, and produces a summary — all without intermediate data flowing through the model.

churn-analysis.sh — agent-generated script 
 1 #!/bin/bash
 2 # Agent writes this script, mcpd executes it.
 3 # 50+ API calls across 2 servers. Zero tokens wasted.
 4
 5 # Step 1: Find the right tool on PostHog
 6 $ mcpd grep "query" --json \
 7   | jq -r '.[] | select(.server=="posthog") | .tool.name'
 8   query-run
 9
10 # Step 2: Query churned customers from PostHog (last 30 days)
11 $ mcpd call posthog/query-run '{
12   "query": {
13     "kind": "HogQLQuery",
14     "query": "SELECT person_id, person.properties.email,
15       person.properties.name, person.properties.company
16       FROM events WHERE event = '\''$churn'\''
17       AND timestamp > now() - interval 30 day"
18   }
19 }' --json \
20   | jq '.content[0].text | fromjson | .results' \
21   > /tmp/churned.json
22
23 # Step 3: For each churned customer, pull their Intercom conversations
24 $ jq -r '.[].email' /tmp/churned.json | while read email; do
25   mcpd call intercom/search-conversations \
26     "{\"query\":\"$email\",\"limit\":5}" --json \
27     | jq '.content[0].text' \
28     >> /tmp/churn_conversations.jsonl
29 done
30
31 # Step 4: Combine into a single report for the LLM
32 $ jq -s '{
33   churned_count: (. | length),
34   customers: [.[] | {email, company, last_messages}]
35 }' /tmp/churn_conversations.jsonl \
36   > /tmp/churn_report.json
37
38 # Only the final 2KB summary enters the model — not the
39 # 500KB of raw PostHog events and Intercom transcripts.

Or use the Node.js API

mcpd is also an npm package. Agents can write Node.js scripts that import mcpd and call tools programmatically — with typed results, async/await, and the full npm ecosystem at their disposal.

churn-analysis.mjs — agent-generated skill 
 1 import { createClient } from 'mcpd';
 2
 3 const client = await createClient();
 4
 5 // Step 1: Query churned customers from PostHog
 6 const churn = await client.call('posthog/query-run', {
 7   query: {
 8     kind: 'HogQLQuery',
 9     query: `SELECT person_id, person.properties.email,
10       person.properties.name, person.properties.company
11       FROM events WHERE event = '$churn'
12       AND timestamp > now() - interval 30 day`
13   }
14 });
15
16 const customers = JSON.parse(churn.content[0].text).results;
17
18 // Step 2: Pull support history from Intercom (in parallel)
19 const conversations = await Promise.all(
20   customers.map(c =>
21     client.call('intercom/search-conversations', {
22       query: c.email, limit: 5
23     })
24   )
25 );
26
27 // Step 3: Build the summary — this is all the LLM sees
28 const report = customers.map((c, i) => ({
29   email: c.email,
30   company: c.company,
31   messages: JSON.parse(conversations[i].content[0].text)
32 }));
33
34 console.log(JSON.stringify({ churned: report.length, customers: report }));

For agents in production: Dedicated block explaining the skills-over-tools advantage. Visual: a simple before/after comparison showing context window allocation. This is the pitch for teams shipping real agents.

When tools leave, skills win

Models execute skills — procedural knowledge in the system prompt — with significantly higher accuracy than raw tool calls. By offloading tools to mcpd, your agent's context window reallocates from tool schemas to the instructions that actually drive quality.

For agents in production, this means measurably better results on complex workflows without changing a single prompt.

Installation

Install mcpd globally via npm, or use it directly with npx. Requires Node.js 20 or later.

terminal 
$ npm install -g mcpd

Once installed, any mcpd command will auto-start the background daemon if it's not already running. The daemon shuts down automatically after 5 minutes of inactivity.

Create a mcpd.config.json in your project root to configure which MCP servers to connect to. The format is compatible with the mcpServers section of claude_desktop_config.json.
