# Typegen

> Generate TypeScript types from live MCP tool schemas so `call()` gets typed arguments and return values.

## Motivation

The JS API (`07-js-api.md`) gives you `call(name: string, args?: Record<string, unknown>)` — fully untyped. If you call `filesystem/read_file`, you get no autocomplete on `args` and no type on the result. You're on your own.

MCP tools already declare their shapes via `inputSchema` (required) and `outputSchema` (optional) as JSON Schema objects. Typegen introspects the running daemon, converts those schemas to TypeScript types, and produces a type map that makes `call()` fully typed — autocomplete on tool names, typed arguments, typed results.

This is the same model as Prisma: run a generate command, types appear in `node_modules`, your editor lights up. No extra imports, no generated files in your source tree.

## How It Works

```
mcpd typegen
     │
     ▼
  daemon (tools/list)
     │
     ▼
  json-schema-to-typescript
     │
     ▼
  node_modules/mcpd/mcpd.generated.d.ts
     │
     ▼
  declare module 'mcpd' { interface McpdToolMap { ... } }
     │
     ▼
  createClient() returns typed client — no extra imports
```

The key mechanism is **TypeScript declaration merging**. The `mcpd` package ships an empty `McpdToolMap` interface. The generated file uses `declare module 'mcpd'` to augment it with tool entries. TypeScript merges the two declarations, and `createClient()` — which is generic over `McpdToolMap` — picks up the merged type automatically.

## CLI Command

```
mcpd typegen [--config <path>]
```

| Flag | Default | Description |
|------|---------|-------------|
| `--config`, `-c` | (default resolution) | Path to `mcpd.config.json` |

The command:

1. Ensures the daemon is running (auto-starts if needed, same as all CLI commands)
2. Calls `tools/list` to fetch all tools from all servers
3. Converts each tool's `inputSchema` and `outputSchema` to TypeScript types
4. Writes `node_modules/mcpd/mcpd.generated.d.ts`

```bash
# Generate types into node_modules
mcpd typegen

# After changing mcpd.config.json or updating servers
mcpd typegen
```

No `--output` flag. The file always goes to `node_modules/mcpd/mcpd.generated.d.ts` — that's the point. Same as `prisma generate` always writes to `node_modules/.prisma/client`.

## Base Package Types

The `mcpd` package ships these types (in `src/client/index.ts`):

```ts
/** Empty by default — augmented by `mcpd typegen`. */
export interface McpdToolMap {}

type HasTools = keyof McpdToolMap extends never ? false : true;

export interface McpdClient {
  // When McpdToolMap is populated, call() is typed.
  // When empty, falls back to untyped.
  call<K extends string>(
    name: HasTools extends true ? (K extends keyof McpdToolMap ? K : K & {}) : string,
    args: K extends keyof McpdToolMap ? McpdToolMap[K]['input'] : Record<string, unknown>,
    options?: CallOptions
  ): Promise<K extends keyof McpdToolMap ? McpdToolMap[K]['output'] : CallResult>;

  // ... other methods unchanged
}
```

When `McpdToolMap` is empty (no typegen), `call()` accepts `(string, Record<string, unknown>)` — same as before. When populated, it narrows to known tool names with typed args/output while still accepting unknown tools via the `K & {}` trick.

## Generated Output

For a daemon with two servers (`filesystem` with `read_file`/`write_file`, `github` with `create_issue`):

```ts
// node_modules/mcpd/mcpd.generated.d.ts
// Auto-generated by `mcpd typegen` — do not edit
// Run `mcpd typegen` to regenerate

declare module 'mcpd' {
  interface McpdToolMap {
    'filesystem/read_file': {
      input: {
        path: string;
        encoding?: string;
      };
      output: unknown;
    };
    'filesystem/write_file': {
      input: {
        path: string;
        content: string;
      };
      output: unknown;
    };
    'github/create_issue': {
      input: {
        repo: string;
        title: string;
        body?: string;
        labels?: string[];
      };
      output: {
        number: number;
        url: string;
      };
    };
  }
}

export {};
```

The `declare module 'mcpd'` block merges with the base `McpdToolMap` interface. The `export {}` makes it a module (required for ambient module declarations to work in a `.d.ts` file).

## User Experience

### Before typegen

```ts
import { createClient } from 'mcpd';

const mcpd = await createClient();
const result = await mcpd.call('filesystem/read_file', { path: '/etc/hostname' });
//                              ^ string                  ^ Record<string, unknown>
// result: CallResult (untyped content array)
```

### After `mcpd typegen`

```ts
import { createClient } from 'mcpd';

const mcpd = await createClient();
const result = await mcpd.call('filesystem/read_file', { path: '/etc/hostname' });
//                              ^ 'filesystem/read_file' | 'filesystem/write_file' | 'github/create_issue'
//                                                         ^ { path: string; encoding?: string }
// result: unknown (no outputSchema on this tool)

const issue = await mcpd.call('github/create_issue', { repo: 'foo/bar', title: 'Bug' });
// issue: { number: number; url: string }

// Unknown tools still work — no type error, just untyped
const x = await mcpd.call('some/other_tool', { foo: 'bar' });
// x: CallResult
```

Same import, same `createClient()`, same code — just with types. Nothing in the user's source tree changes.

### When types go stale

If a server adds/removes tools, the generated types are stale. This is fine:

- **New tools**: `call('new/tool', ...)` still works — falls back to untyped. Run `mcpd typegen` to pick up new tools.
- **Removed tools**: `call('old/tool', ...)` type-checks but fails at runtime with "Tool not found". Same as any stale type.
- **Changed schemas**: Arguments may mismatch — the daemon validates against the real schema and returns a clear error.

Run `mcpd typegen` to refresh. Same workflow as `prisma generate`.

## Design Decisions

**`output` defaults to `unknown`.** Most MCP tools don't declare `outputSchema` — they return unstructured `content` arrays. When `outputSchema` is absent, the output type is `unknown`. Users can narrow it themselves or wait for servers to adopt structured output.

**No `createTypedClient`.** Module augmentation means `createClient()` is already typed when `McpdToolMap` is populated. No separate function, no separate import, no cast. This is the whole point of writing to `node_modules`.

**No `--output` flag.** The generated file must be in `node_modules/mcpd/` for module augmentation to work. Letting users put it elsewhere defeats the purpose. If they want a standalone `.d.ts` for a different use case, they can copy it.

**Types disappear on `npm install`.** This is the Prisma tradeoff. `node_modules` is ephemeral. Users need to re-run `mcpd typegen` after installing dependencies. This can be automated with a `postinstall` script:

```json
{
  "scripts": {
    "postinstall": "mcpd typegen"
  }
}
```

**Overloaded `call()` with fallback.** The type-level conditional `K extends keyof McpdToolMap ? ... : ...` ensures unknown tool names still compile. Stale types degrade to untyped, they don't cause type errors.

## JSON Schema to TypeScript Conversion

Use `json-schema-to-typescript` for the conversion. It handles:

- `type: "string" | "number" | "integer" | "boolean" | "null"` → TS primitives
- `type: "object"` with `properties` / `required` → interfaces with optional fields
- `type: "array"` with `items` → `T[]`
- `enum` → union literals
- `oneOf` / `anyOf` → union types
- `allOf` → intersection types
- `$ref` → resolved inline (MCP tool schemas rarely use `$ref`)
- Nested objects → nested interfaces

### Edge Cases

| JSON Schema | TypeScript | Notes |
|---|---|---|
| No `outputSchema` | `unknown` | Most tools today |
| `properties: {}` or missing `properties` | `Record<string, unknown>` | Accept anything |
| `additionalProperties: true` | `{ [k: string]: unknown; ...named }` | Indexed signature + named props |
| Unsupported keywords | Ignored, falls back to `unknown` | Graceful degradation |
| Empty `inputSchema` (no `properties`, no `required`) | `Record<string, never>` | Tool takes no arguments |

### Naming

Types are generated inline in the tool map — no separate named interfaces. This avoids name collisions (two servers could have tools with the same name) and keeps the output self-contained within the `declare module` block.

### `src/codegen/typegen.ts`

```ts
import { compile } from 'json-schema-to-typescript';

type ToolEntry = {
  server: string;
  tool: {
    name: string;
    inputSchema: Record<string, unknown>;
    outputSchema?: Record<string, unknown>;
  };
};

export async function generateTypes(tools: ToolEntry[]): Promise<string> {
  const toolTypes: string[] = [];

  for (const { server, tool } of tools) {
    const qualifiedName = escapeKey(`${server}/${tool.name}`);
    const inputType = await schemaToType(tool.inputSchema, `${server}_${tool.name}_input`);
    const outputType = tool.outputSchema
      ? await schemaToType(tool.outputSchema, `${server}_${tool.name}_output`)
      : 'unknown';

    toolTypes.push(
      `    '${qualifiedName}': {\n      input: ${indent(inputType, 6)};\n      output: ${indent(outputType, 6)};\n    };`
    );
  }

  return `// Auto-generated by \`mcpd typegen\` — do not edit
// Run \`mcpd typegen\` to regenerate

declare module 'mcpd' {
  interface McpdToolMap {
${toolTypes.join('\n')}
  }
}

export {};
`;
}

async function schemaToType(schema: Record<string, unknown>, name: string): Promise<string> {
  const compiled = await compile(schema, name, {
    bannerComment: '',
    additionalProperties: false,
    unknownAny: true,
  });
  return extractTypeBody(compiled);
}

function extractTypeBody(compiled: string): string {
  const match = compiled.match(/\{[\s\S]*\}/);
  if (!match) return 'Record<string, unknown>';
  return match[0];
}

function escapeKey(key: string): string {
  return key.replace(/\\/g, '\\\\').replace(/'/g, "\\'");
}

function indent(text: string, spaces: number): string {
  const pad = ' '.repeat(spaces);
  return text.split('\n').join('\n' + pad);
}
```

### `src/cli/commands/typegen.ts`

```ts
import { sendRequest, ensureDaemon } from '../client.js';
import { generateTypes } from '../../codegen/typegen.js';
import fs from 'node:fs';
import path from 'node:path';
import { createRequire } from 'node:module';

export async function typegenCommand(options: { config?: string }) {
  await ensureDaemon(options.config);

  const tools = await sendRequest('tools/list');

  const content = await generateTypes(tools as ToolEntry[]);

  // Resolve node_modules/mcpd/ from the user's project
  const require = createRequire(path.resolve('package.json'));
  const mcpdPkgDir = path.dirname(require.resolve('mcpd/package.json'));
  const outputPath = path.join(mcpdPkgDir, 'mcpd.generated.d.ts');

  fs.writeFileSync(outputPath, content, 'utf-8');

  console.log(`Generated ${(tools as unknown[]).length} tool types → ${outputPath}`);
}
```

In `src/cli/index.ts`:

call<K extends string>(

### Package `types` Field

Ensure `mcpd.generated.d.ts` is picked up by adding a reference in the package's type declarations:

```ts
// In the package's main .d.ts (e.g. dist/client.d.mts), add:
/// <reference path="./mcpd.generated.d.ts" />
```

This triple-slash reference tells TypeScript to include the generated file when the `mcpd` module is loaded, even though nothing explicitly imports it.

Also ship a stub so TypeScript doesn't error before typegen runs:

```ts
// node_modules/mcpd/mcpd.generated.d.ts (shipped with package)
// Stub — run `mcpd typegen` to populate with tool types
export {};
```

## Testing

### Unit Tests (`src/codegen/typegen.test.ts`)

```ts
describe('generateTypes', () => {
  it('generates module augmentation with tool types', async () => {
    const tools = [
      {
        server: 'fs',
        tool: {
          name: 'read',
          inputSchema: {
            type: 'object',
            properties: { path: { type: 'string' } },
            required: ['path'],
          },
        },
      },
    ];
    const output = await generateTypes(tools);
    expect(output).toContain("declare module 'mcpd'");
    expect(output).toContain("'fs/read'");
    expect(output).toContain('path: string');
    expect(output).toContain('output: unknown');
  });

  it('generates typed output when outputSchema present', async () => {
    const tools = [
      {
        server: 'gh',
        tool: {
          name: 'create_issue',
          inputSchema: {
            type: 'object',
            properties: { title: { type: 'string' } },
            required: ['title'],
          },
          outputSchema: {
            type: 'object',
            properties: { number: { type: 'integer' }, url: { type: 'string' } },
            required: ['number', 'url'],
          },
        },
      },
    ];
    const output = await generateTypes(tools);
    expect(output).toContain('number: number');
    expect(output).toContain('url: string');
    expect(output).not.toContain('output: unknown');
  });

  it('handles tools with no properties', async () => {
    const tools = [
      {
        server: 'srv',
        tool: {
          name: 'ping',
          inputSchema: { type: 'object' },
        },
      },
    ];
    const output = await generateTypes(tools);
    expect(output).toContain("'srv/ping'");
  });

  it('handles empty tool list', async () => {
    const output = await generateTypes([]);
    expect(output).toContain('interface McpdToolMap');
    expect(output).toContain('export {}');
  });

  it('escapes special characters in tool names', async () => {
    const tools = [
      {
        server: 'my-server',
        tool: {
          name: "tool's",
          inputSchema: { type: 'object' },
        },
      },
    ];
    const output = await generateTypes(tools);
    expect(output).toContain("my-server/tool\\'s");
  });
});
```
