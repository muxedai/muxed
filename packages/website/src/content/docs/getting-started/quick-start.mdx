## Three commands to your first tool call

### 1. Generate a config

```bash
npx muxed init
```

`init` scans your machine for existing MCP agent configurations — Claude Desktop, Cursor, VS Code, and others — and writes a `muxed.config.json` file with every server it finds. No manual setup required.

### 2. List available tools

```bash
muxed tools
```

This prints every tool exposed by every configured server, with names and one-line descriptions.

### 3. Call a tool

```bash
muxed call filesystem/read_file '{"path": "/tmp/config.json"}'
```

That's it. The tool runs through the daemon and only the result comes back.

## Manual configuration

If you prefer to create a config by hand instead of using `init`, add a `muxed.config.json` in your project root:

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"],
      "env": {}
    }
  }
}
```

Each key under `mcpServers` is a server name. The `command` and `args` fields tell muxed how to start the server process.

## Discover and call tools

Once your servers are configured, use `grep`, `info`, and `call` to find and invoke tools without loading every schema into your agent's context.

```bash
# Search for tools by name or description
muxed grep "read"
#   filesystem/read_file      Read file contents
#   postgres/query            Read data via SQL

# Inspect a tool's schema
muxed info filesystem/read_file
#   path (string, required)   File path to read

# Call it — only the result enters the LLM context
muxed call filesystem/read_file '{"path": "/tmp/config.json"}'
```

This is the core workflow: search, inspect, call. Agents follow the same pattern — they pull in only the tools they need for the current step instead of loading everything up front.

## What happened behind the scenes

- **The daemon started automatically** on the first command. There is nothing to launch or manage.
- **It connected to the configured MCP servers** and cached their tool schemas in memory.
- **`grep` searched tool names and descriptions** across all servers, without any of those schemas entering the LLM context.
- **`call` invoked the tool and returned only the result** — the raw output, ready for the agent to use.
